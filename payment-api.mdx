Finverse's Payments API allow you to:

- **Collections: Set up single and recurring payments (collections)**, as well as direct debit mandates, from end-users' accounts at supported institutions
- **Payouts: Setup payouts from Finverse** to end-users' accounts at supported institutions.

### **Collections**

Finverse supports two collections flows:

- **A. Payment Links (recommended integration)**: our easy-to-use payment links allow end-users to select a payment method and confirm a payment directly in Finverse's UI. Finverse will also provide email notifications to end-users throughout the payment journey (e.g. when a Mandate is successfully setup, or a FPS payment is received). Payment Links support multiple payment methods:
  - Direct debit mandates (enabled by default)
  - FPS QR code (enabled by default)
  - Credit cards (disabled by default -- contact Finverse to enable).
- **B. Mandate-based Payments**: payments are executed based on a pre-authorized Mandate (e.g. Direct Debit mandate). This allows customers to separate the Mandate setup and Payment steps (e.g. to setup a Mandate upfront and collect a Payment 1 month later).

### **Payouts**

Finverse currently supports scheduled payouts, funded by either:

- Direct debit mandates you have setup
- Your current funds balance

Please contact Finverse to enable access to Payouts and related endpoints (Payment Users, Payment Accounts).

# **Get Started**

After Finverse API authentication, follow these steps to collect a Payment from your first Institution (e.g. Testbank).

#### **A. Payment Links**

- Create a Payment Link URL via `POST /payment_links`
- Invite the end-user to setup their Payment in the Finverse UI (using the Payment Link URL generated). The end-user will be asked to select their payment method, provide their payment details, and confirm the payment.
- Check the Payment Link until the Payment Link status is `PAID` (via webhooks or `GET /payment_links/{payment_link_id}`)

#### **B. Mandate-based Payment**

- Create a Mandate via `POST /mandates`
- Generate a Mandate `link_url` using the `mandate_id` via `POST /mandates/link`
- Invite the end-user to authorize the Mandate in the Finverse UI (using the Mandate `link_url` generated). The end-user will be asked to confirm their account details.
- Check the Mandate until the Mandate status is `SUCCEEDED` (via webhooks or `GET /mandates/{mandate_id}`)
- Create a Payment via `POST /payments`
- Check the Payment until the Payment status is `EXECUTED` (via webhooks or `GET /payments/{payment_id}`)

Amount to be paid in currency's smallest unit. For example, HKD 100.01 is represented as amount = 10001. For currencies without minor units (e.g. VND, JPY), the amount is represented as is, without modification. For example, VND 15101 is represented as amount = 15101.

## **Sequence diagrams: Payments API**

#### **Payment Links: Payment mode**

This flow is used to collect a one-time Payment from a user using an online link, with additional options to store the user's Payment Method and setup future autopayments.

```mermaid placement="top-right"
sequenceDiagram
    title Finverse Payment Links - online payments flow

    box Merchant
        actor User
        participant MerchantUI as Merchant UI
        participant MerchantBE as Merchant backend/ERP
    end

    box Finverse
        participant FPUI as Finverse Payment UI
        participant FAPI as Finverse API
        participant Gateways as Payment Gateways
    end

    Note over User, MerchantBE: Payment/invoice preflow (merchant-specific)

    Note over MerchantBE, FAPI: API authentication - api.prod.finverse.net

    MerchantBE->>FAPI: [1] POST /auth/customer/token (client_id, client_secret)
    FAPI-->>MerchantBE: customer_token

    Note over MerchantBE, FAPI: Payment Link creation - mode=payment

    MerchantBE->>FAPI: [2] POST /payment_links (amount, external_user_id, unique_reference_id, metadata)
    FAPI-->>MerchantBE: Payment Link (url, payment_link_id)

    MerchantBE-->>MerchantUI: Show Pay Now button with url

    User->>FPUI: Open url - Pay Now (Finverse Payment UI)

    Note over FPUI, Gateways: User selects payment method and pays (banks/cards). Some methods can take up to 48h to confirm.

    FPUI-->>MerchantBE: HTTP 302 redirect to merchant redirect_url (ui_mode=redirect)

    Note over MerchantBE, FAPI: Async handling - webhooks + GET for Payment Link status

    FAPI-->>MerchantBE: [3a] Payment Link and Payment webhooks

    MerchantBE->>FAPI: [3b] GET /payment_links/payment_link_id (fallback + details)
    FAPI-->>MerchantBE: Payment Link (session_status, payment, payment_method)

    Note right of MerchantBE: Record Payment and Payment Method for future payments

    Note over MerchantUI, MerchantBE: Show outcome from session_status

    alt session_status OPEN
        MerchantBE-->>MerchantUI: Show Pay Now with url
    else session_status COMPLETE
        MerchantBE-->>MerchantUI: Show payment success
    else session_status FAILED
        MerchantBE-->>MerchantUI: Show failure + retry url
    else session_status PROCESSING
        MerchantBE-->>MerchantUI: Show processing - notify later
    end

    Note over User, MerchantBE: Async notifications

    alt session_status COMPLETE
        MerchantBE-->>User: Notify payment success
    else session_status FAILED
        MerchantBE-->>User: Notify failure + retry url
    end
```

This flow is used to allow users to setup a direct debit authorization mandate (i.e. eGIRO/eDDA setup), which can then be used to collect future payments (using the Payment collection flow below).

```mermaid placement="top-right"
sequenceDiagram
    title Finverse direct debits - mandate setup flow

    box Merchant
        actor User
        participant MerchantUI as Merchant UI
        participant MerchantBE as Merchant backend/ERP
    end

    box Finverse
        participant FMUI as Finverse Mandate Setup UI
        participant FAPI as Finverse API
        participant BankAPI as Bank direct debit APIs
    end

    Note over User, MerchantBE: Mandate setup preflow (merchant-specific)

    Note over MerchantBE, FAPI: API authentication - api.prod.finverse.net

    MerchantBE->>FAPI: [1] POST /auth/customer/token (client_id, client_secret)
    FAPI-->>MerchantBE: customer_token

    Note over MerchantBE, FAPI: Mandate creation

    MerchantBE->>FAPI: [2] POST /mandates (external_user_id, user_type, recipient_account_id, metadata)
    FAPI-->>MerchantBE: Mandate object (mandate_id, payment_method_id, status AUTHORIZATION_REQUIRED)

    MerchantBE->>FAPI: [3] POST /mandates/link (mandate_id, link_customizations)
    FAPI-->>MerchantBE: Mandate Link URL (link_url, 1hr expiry)

    MerchantBE-->>MerchantUI: Show Setup Mandate button with url

    User->>FMUI: Open url - Setup Mandate (Finverse Mandate Setup UI)

    Note over FMUI, BankAPI: User selects bank and completes authorization - SG redirects to bank, HK in Finverse UI - up to 48h for some business accounts

    FMUI-->>MerchantBE: HTTP 302 redirect to merchant redirect_url (ui_mode=redirect)

    Note over MerchantBE, FAPI: Async handling - webhooks + GET for Mandate status

    FAPI-->>MerchantBE: [4a] Mandate webhooks

    MerchantBE->>FAPI: [4b] GET /mandates/mandate_id (fallback if webhooks not received)
    FAPI-->>MerchantBE: Mandate object (mandate_id, payment_method_id, status SUCCEEDED or FAILED)

    Note right of MerchantBE: Record Mandate for future payments

    Note over MerchantUI, MerchantBE: Show Mandate setup result from status

    alt status AUTHORIZATION_REQUIRED
        MerchantBE-->>MerchantUI: Show Setup Mandate button with url
    else status SUCCEEDED
        MerchantBE-->>MerchantUI: Show mandate success
    else status FAILED
        MerchantBE-->>MerchantUI: Show mandate failed + retry url
    else status PROCESSING or SUBMITTED
        MerchantBE-->>MerchantUI: Show mandate processing - notify later
    end

    Note over User, MerchantBE: Async notifications

    alt status SUCCEEDED
        MerchantBE-->>User: Notify mandate setup success
    else status FAILED
        MerchantBE-->>User: Notify mandate setup failure + retry url
    else status PROCESSING after 2hrs
        MerchantBE-->>User: Mandate still processing - 2nd approver may need to approve via internet banking (business accounts)
    end
```

#### **Payment collection flow (using a stored Payment Method)**

This flow is used by merchants to initiate a Payment collection using a user's stored Payment Method (any stored payment method including card or direct debit mandate).

```mermaid placement="top-right"
sequenceDiagram
    title Finverse Payments - merchant-initiated payment flow

    box Merchant
        actor User
        participant MerchantBE as Merchant backend/ERP
    end

        participant Gateways as Payment Gateways
    end

    Note over MerchantBE: Pre-req - Payment Method setup completed (see Payment Links flow)

    Note right of MerchantBE: Record Payment Method for future payments

    Note over User, MerchantBE: Pre-req - User billing and invoice notifications if needed (merchant-specific)

    Note over MerchantBE: Pre-req - Payment initiation logic - check invoice status and due dates, valid payment method, initiate on due date, retry failed payments (can be handled by Finverse via ERP integration)

    Note over MerchantBE, FAPI: API authentication - api.prod.finverse.net

    MerchantBE->>FAPI: [1] POST /auth/customer/token (client_id, client_secret)
    FAPI-->>MerchantBE: customer_token

    Note over MerchantBE, FAPI: Payment initiation

    MerchantBE->>FAPI: [2] POST /payments (amount, payment_method_id)
    FAPI-->>MerchantBE: Payment (status PROCESSING, payment_id)

    Note over FAPI, Gateways: Finverse payment processing - near real-time, usually under 5s

    Note over MerchantBE, FAPI: Payment status check - webhooks + GET to resolve result

    FAPI-->>MerchantBE: [3a] Payment webhooks

    MerchantBE->>FAPI: [3b] GET /payments/payment_id (fallback if webhooks not received)
    FAPI-->>MerchantBE: Payment (status)

    Note right of MerchantBE: Record Payment

    Note over User, MerchantBE: User payment notifications based on Payment status (can be handled by Finverse)

    alt status EXECUTED
        MerchantBE-->>User: Notify payment success
    else status FAILED
        MerchantBE-->>User: Notify payment failure + Payment Link url to retry or change payment method
    end
```

# **Implementation notes**

#### **Currency amounts: "Minor" units**

**All amounts in the Payments API are expressed as integers in ISO 4217 "minor" currency units.**

A "minor" currency unit represents an amount in a currency's smallest unit (usually, but not always, "cents"). For example, HKD 100.01 is represented as amount = 10001.

For currencies without minor units (e.g. VND, JPY), the amount is represented as is, without modification. For example, VND 15101 is represented as amount = 15101.

#### **Idempotent requests**

Certain `POST` requests require an `Idempotency-Key` header. This makes it safe to retry the request without re-triggering the underlying operation. Any retries (within 24 hours) will return the same response as the first request using the same key.

Idempotent requests work by saving the response of the first request made for the idempotency key (after the request passes validation). Subsequent requests with the same key return the same result.

The idempotency key should be a unique key with enough entropy to avoid collisions. We recommend using UUIDs, ULIDs or similar random strings.
